# Hackathon Pitch Packager

Use this skill to translate implementation work into a short, credible story that judges can score quickly.

## Workflow

1. Identify the one-sentence problem.
2. Identify the one-sentence solution.
3. Choose one polished live demo flow.
4. Map the build directly to judging criteria.
5. Show where Codex was central, not merely helpful.

## Packaging Rules

- Lead with the problem and the user, not the tech stack.
- Show one domain clearly, then mention reuse into adjacent domains.
- Use visible proof from the product: live state, logs, confidence, and commands.
- Avoid claiming production readiness unless the prototype actually demonstrates it.
- Keep the Codex story concrete: generation, iteration, debugging, refinement.

## Judging Alignment

Read [references/judging-checklist.md](references/judging-checklist.md) before drafting a pitch, demo script, or summary.

## Deliverables

When asked to package a prototype, produce:

- a one-line problem statement
- a one-line solution statement
- a 60 to 90 second demo script
- a criteria mapping
- 3 to 5 proof points showing Codex at the center
- a short “what comes next” adoption story

## Demo Guidance

- Start from an idle state.
- Trigger one voice command early to establish the multimodal concept.
- Show one or two gestures clearly.
- End on the evidence log or architecture explanation.
- Prefer a crisp story over feature count.
